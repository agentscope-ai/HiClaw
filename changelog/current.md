- fix(manager): start Higress WASM plugin-server (`/usr/share/nginx/html/plugins/...` on port 8002) from the same nginx instance as Element Web. Our supervisord configs (both embedded and legacy) override the base image's supervisord which previously launched a separate `plugin-server` program â€” without it Envoy cannot fetch `ai-proxy`/`key-auth`/`ai-statistics` WASM modules and forwards LLM requests to upstream without Host header rewrite, resulting in 404s from the LLM backend. Listens on both IPv4 and IPv6 because Envoy resolves `localhost` to `::1` first.

- fix(worker): stop Worker-side sync from pushing `SOUL.md`, `AGENTS.md`, and `HEARTBEAT.md` back to MinIO. These files are authoritatively written by the controller (via inline `spec.soul`/`spec.agents` or package contents); the conditional `mc cp`-if-newer loop in `worker-entrypoint.sh` raced with controller pushes â€” for example `hiclaw apply -f` of a Worker with inline overrides would be reverted by a stale Worker push shortly after, leaving MinIO with the ORIGINAL package content. The corresponding `mc mirror` exclude list already covered these files; the individual re-push was added speculatively for "personality evolution" but no code path actually writes these files from the Worker side.

- chore(manager): drop the `hiclaw-controller` istiod debug push workaround (`GET 127.0.0.1:15014/debug/adsz?push=true`) and its `gateway.Config.PilotURL` plumbing. The original Higress all-in-one image had a bug where Console writes were not promptly pushed to Envoy in embedded mode, so the controller manually kicked istiod after consumer/route updates; this is fixed in the pinned `higress/all-in-one:2.2.1` base, and the manual trigger is no longer needed.

- fix(manager): always exclude `SOUL.md`, `AGENTS.md`, and `HEARTBEAT.md` from the initial localâ†’OSS mirror in `Deployer.DeployWorkerConfig`. Each of these files has a dedicated authoritative writer (PutObject for SOUL.md, `prepareAndPushAgentsMD` for AGENTS.md, `pushBuiltinTopLevelFiles` for HEARTBEAT.md). Previously the mirror only excluded SOUL.md/AGENTS.md when the spec carried inline overrides; on subsequent reconciles (e.g. finalizer add or status bump) the mirror re-pushed the stale local AGENTS.md (wrapped with empty builtin markers by `executor.DeployToMinIO`) over the correctly-merged version that `prepareAndPushAgentsMD` had just written, transiently exposing missing `## Every Session` content and a missing team-context block. This is the root cause of `test-17-worker-config-verify` flakes where assertions on AGENTS.md content failed right after worker creation but passed after a subsequent update operation.

- fix(worker): restore the per-file `mc cp`-if-newer push-back of `SOUL.md`, `AGENTS.md`, and `HEARTBEAT.md` in `worker-entrypoint.sh` that commit 394a7d3 deleted. The deletion was based on the claim that "no code path actually writes these files from the Worker side", but the agent itself is the writer for the documented self-edit use cases â€” HEARTBEAT.md checklist tweaks (`worker-agent/AGENTS.md`: "you are free to edit HEARTBEAT.md") and SOUL.md "personality evolution". Without push-back those edits were silently lost on container restart or any subsequent file-sync pull. The original `-nt ${PULL_MARKER}` guard is sufficient on its own: every pull path (startup mirror, retry mirror, file-sync skill) touches `${PULL_MARKER}` strictly after `mc mirror` returns, so freshly-pulled package content always has mtime â‰¤ marker and only post-pull agent edits trigger the push. Also fix the 5-minute Remoteâ†’Local fallback loop which forgot to `touch ${PULL_MARKER}` after pulling `openclaw.json`/`skills/`, causing the change-triggered push loop's `find -newer` trigger to fire on every iteration.

- fix(manager): write `channels.matrix.userId = "@manager:${HICLAW_MATRIX_DOMAIN}"` in both `manager-openclaw.json.tmpl` and the in-place `start-manager-agent.sh` jq update path, so OpenClaw's matrix-auto-reply does not fall back to the `@default` localpart and silently drop incoming DM messages from the admin user

- fix(controller): re-pin `WorkerName: "manager"` in `DeployManagerConfig` so the generated Manager `openclaw.json` always uses `userId=@manager:<domain>` regardless of the Manager CR name (typically `default`). The controller refactor in `d029ed2` accidentally reverted commit `3f8f84b`, which made the controller-pushed `openclaw.json` overwrite the manager-rendered one with `userId=@default:<domain>` shortly after startup. OpenClaw's Matrix client then filtered every admin DM to the wrong localpart, so the agent silently received zero messages and `test-02-create-worker` failed with `LLM Calls: 0` even though the gateway, sync and access token were healthy.

- fix(controller): write `channels.matrix.network.dangerouslyAllowPrivateNetwork = true` into every controller-generated `openclaw.json` (workers and manager). The `openclaw` 2026.4.x bump rolled the matrix SSRF fix forward but the new auto-detection only treats `localhost`, raw IP literals, single-label hostnames, and `.local`/`.internal` suffixes as private â€” the embedded homeserver is reached via `matrix-local.hiclaw.io:18080` (a public-looking FQDN that resolves to `127.0.0.1`) and k8s workloads via cluster service DNS, so neither path opts into private-network access by default. Without this flag every freshly-provisioned worker fails to `/sync` against the homeserver, never reaches `PREPARED`, never joins its room and never replies â€” surfaced in CI as `test-15-import-worker-zip` "Worker did not join room within 120s" and as `test-02/03/04/06` "Manager replied (value is empty or null)" once the same regression hit the manager-side controller-rewritten config.
- fix(controller+manager): restore OpenClaw Control UI access at host port 18888 after the openclaw 2026.4.x upgrade.
  - **Port unification**: openclaw 2026.4.x merged the Control UI HTTP server into the gateway port (older versions ran a separate Control UI listener on 18799 alongside the gateway WS on 18800). The controller's hardcoded `gateway.port: 18800` left the new unified listener on 18800 while every downstream consumer â€” Manager Dockerfile `EXPOSE 18799`, the host:`HICLAW_PORT_MANAGER_CONSOLE`â†’container:18799 port mapping in `manager_reconcile_container.go`, the legacy nginx reverse proxy in `start-element-web.sh`, and the `tests/lib/test-helpers.sh` / `install/hiclaw-verify.sh` health probes â€” kept addressing 18799 and got connection-refused. Pin `gateway.port: 18799` in `agentconfig/generator.go`, `manager-openclaw.json.tmpl`, the in-place `start-manager-agent.sh` jq merge, and the worker template skill reference.
  - **Bind exposure**: openclaw 2026.4.x also flipped the default bind address from `0.0.0.0` to `127.0.0.1`. In hiclaw's embedded dual-container mode the manager runs in its own container and the host port mapping (host:18888 â†’ manager-container:18799) cannot reach a loopback-only listener inside the container â€” connections were accepted by the container runtime's port forwarder and immediately reset because nothing was actually listening on the container's interface. Set `gateway.bind = "lan"` so the gateway binds to `0.0.0.0`; access remains gated by the shared `gateway.auth.token`.
  - **Browser auth gates**: with non-loopback binds, openclaw 2026.4.x also requires explicit `gateway.controlUi.allowInsecureAuth = true` (HTTP-only console) and `gateway.controlUi.allowedOrigins = ["*"]` (hiclaw cannot enumerate every legitimate origin since the user picks `HICLAW_PORT_MANAGER_CONSOLE` and may reach the console via 127.0.0.1, LAN IP, or custom hostname). Add both alongside the existing `dangerouslyDisableDeviceAuth = true` flag, and re-emit the full controlUi block in the controller-pushed config so `mc mirror` no longer strips it.
